Below is a practical list of **Blackwell-specific tuning levers** worth using when writing **custom CUDA/PTX kernels for LLM inference on an RTX 5060 Ti 16 GB**, not llama.cpp/TensorRT/etc.

The RTX 5060 Ti is a **Blackwell, compute capability 12.0** GPU, and NVIDIA lists it under CC 12.0 workstation/consumer devices. It has **4608 CUDA cores**, **5th-gen Tensor Cores**, **16 GB or 8 GB GDDR7**, and a **128-bit memory interface**. ([NVIDIA Developer][1]) ([NVIDIA][2])

## Highest-impact custom-kernel tunes

| Tune                                                                                  | What to do in a custom LLM kernel                                                                                                                                                                                                       | Why it matters on RTX 5060 Ti / Blackwell                                                                                                          |
| ------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Target SM_120 explicitly**                                                          | Compile native cubins for Blackwell consumer: `-gencode=arch=compute_120,code=sm_120`, and optionally include PTX fallback. CUDA 12.8 added compiler support for **SM_100, SM_101, and SM_120**. ([NVIDIA Docs][3])                     | Avoid relying only on PTX JIT. RTX 5060 Ti is CC 12.0, not Hopper/Ada.                                                                             |
| **Use FP4 E2M1 / NVFP4 where accuracy allows**                                        | Quantize weights, and possibly KV/value paths, to FP4 E2M1 with scale factors. CUDA exposes `__nv_fp4_e2m1`, `__nv_fp4x2_e2m1`, and `__nv_fp4x4_e2m1` storage/conversion types. ([NVIDIA Docs][4])                                      | Blackwell 5th-gen Tensor Cores expose FP4 capability; this is the biggest theoretical LLM throughput and memory-bandwidth win.                     |
| **Use block-scaled FP4 MMA PTX**                                                      | For warp-level kernels, look at PTX forms like `mma.sync.aligned.m16n8k64.row.col.kind::mxf4.block_scale.f32.e2m1.e2m1.f32.ue8m0` and `kind::mxf4nvf4.block_scale.scale_vec::4X...`. ([NVIDIA Docs][5])                                 | This is the relevant low-level path for custom Blackwell FP4 GEMM/GEMV-style kernels. Accumulate in FP32 or FP16 depending on quality/perf target. |
| **Use FP8 / FP6 / mixed FP formats when FP4 quality is too low**                      | PTX supports mixed low-precision MMA kinds involving FP8/FP6/FP4, for example `kind::f8f6f4` forms using `.e2m1`, `.e2m3`, etc. ([NVIDIA Docs][5])                                                                                      | For LLMs, FP4 is not universally safe. FP8 or mixed FP8/FP4 can be a better quality/performance tradeoff.                                          |
| **Exploit packed vector types**                                                       | Store sub-byte data packed: FP4x2 / FP4x4 style packing, aligned vectorized loads, and avoid unpacking to scalar floats too early. CUDA Math API exposes packed FP4 structs. ([NVIDIA Docs][4])                                         | RTX 5060 Ti has only a 128-bit memory interface, so bandwidth-efficient packing is crucial.                                                        |
| **Keep global memory accesses coalesced**                                             | Lay out Q/K/V, weights, scales, and output tiles so each warp loads contiguous 32-byte aligned segments. NVIDIA explicitly recommends coalesced global accesses as a high-priority CUDA best practice. ([NVIDIA Docs][6])               | LLM inference is often memory-bound, especially GEMV/decode. Poor coalescing can erase FP4 gains.                                                  |
| **Use shared-memory tiling, but respect the 99 KB/block limit**                       | For CC 12.0, NVIDIA lists **128 KB shared memory per SM** and **99 KB max shared memory per thread block**. ([NVIDIA Docs][6])                                                                                                          | Don’t blindly port B200/SM100 kernels that assume much larger per-block shared memory.                                                             |
| **Tune occupancy around 48 resident warps/SM**                                        | NVIDIA lists **48 concurrent warps per SM** for compute capability 12.0, with 64K 32-bit registers per SM and 255 registers max per thread. ([NVIDIA Docs][6])                                                                          | Attention kernels can become register-bound. More registers may help reuse, but can kill occupancy. Benchmark both.                                |
| **Use `cudaOccupancyMaxActiveClusters` for cluster kernels**                          | If using Thread Block Clusters / Distributed Shared Memory, compute occupancy with NVIDIA’s cluster occupancy API. ([NVIDIA Docs][6])                                                                                                   | Cluster launch choices are architecture-sensitive; guessing cluster size is risky.                                                                 |
| **Use Distributed Shared Memory only for patterns that need inter-block cooperation** | Blackwell supports thread block clusters and DSM; NVIDIA recommends coalesced, 32-byte-aligned DSM accesses and avoiding non-unit stride patterns. ([NVIDIA Docs][6])                                                                   | Potentially useful for fused attention/reductions, but bad DSM patterns can be slower than global/shared alternatives.                             |
| **Use async copy / pipeline staging**                                                 | Use `cuda::pipeline`, `memcpy_async`, or lower-level async copy patterns to overlap global-to-shared movement with compute. CUDA documents hardware-accelerated async copy support and pipeline primitives. ([NVIDIA Docs][7])          | Helps prefill/attention tiles where memory movement and MMA can overlap.                                                                           |
| **Consider TMA only after verifying actual SM120 support and toolchain behavior**     | CUDA docs describe Tensor Memory Accelerator for efficient multidimensional global/shared transfers and list TMA support in feature tables for modern compute capabilities. ([NVIDIA Docs][7])                                          | Useful conceptually for tiled attention/GEMM, but many examples target datacenter Blackwell. Validate on RTX 5060 Ti specifically.                 |
| **Tune L1/shared-memory carveout**                                                    | Use `cudaFuncSetAttribute(..., cudaFuncAttributePreferredSharedMemoryCarveout, ...)` when kernels benefit from more shared memory versus cache. NVIDIA documents runtime-selectable shared/L1 carveout on Blackwell. ([NVIDIA Docs][6]) | Good for tiled attention/GEMM; less shared memory may be better for bandwidth-heavy decode kernels.                                                |
| **Use L2 persistence for hot weights/scales/KV regions**                              | CUDA lets users control L2 persistence for selected data; NVIDIA notes Blackwell retains this capability. ([NVIDIA Docs][6])                                                                                                            | Useful for frequently reused scales, small layers, KV metadata, or repeated decode paths.                                                          |
| **Exploit structured sparsity only if the model is trained/converted for it**         | PTX documents sparse `tcgen05.mma.sp` and sparse granularities, including low-precision kinds such as `.kind::mxf4`. ([NVIDIA Docs][5])                                                                                                 | Can improve matmul throughput, but only if weights obey the required sparsity pattern. Don’t expect arbitrary pruned weights to work.              |
| **Fuse RMSNorm / RoPE / dequant / matmul epilogues**                                  | Fuse dequantization scales, bias, activation, RMSNorm, RoPE, and residual ops into surrounding kernels where possible.                                                                                                                  | Especially important on 16 GB / 128-bit memory systems: each extra kernel launch and global-memory round trip is expensive.                        |
| **Separate prefill and decode kernels**                                               | Prefill is GEMM/attention-tile heavy; decode is GEMV/KV-cache bandwidth heavy. Tune tile sizes, warp mapping, and quantization differently.                                                                                             | One kernel strategy rarely performs well for both phases.                                                                                          |
| **Keep KV cache format bandwidth-first**                                              | Use compact KV formats, grouped-query attention when model supports it, vectorized loads, and page/block layouts that preserve coalescing.                                                                                              | On RTX 5060 Ti, long-context decode will be dominated by KV-cache bandwidth.                                                                       |
| **Use CUDA Graphs for repeated decode launches**                                      | CUDA 12.8 added CUDA Graph conditional execution features and other graph optimizations. ([NVIDIA Docs][3])                                                                                                                             | Reduces launch overhead in token-by-token inference. This is not a kernel instruction, but it matters for custom runtimes.                         |
| **Profile with Nsight Compute on actual RTX 5060 Ti**                                 | Check achieved occupancy, tensor pipe utilization, global load efficiency, shared-memory bank conflicts, register spills, and L2 hit rate.                                                                                              | Blackwell tuning is very architecture-specific; ported Ada/Hopper assumptions are often wrong.                                                     |

## RTX 5060 Ti-specific constraints to keep in mind

The main constraint is **memory bandwidth and capacity**, not just tensor throughput. NVIDIA’s own RTX 5060 Ti spec table lists **16 GB / 8 GB GDDR7** on a **128-bit** bus, so a 16 GB card can hold useful local models, but decode speed will still be sensitive to quantization format, KV-cache layout, and global-memory traffic. ([NVIDIA][2])

Also, don’t blindly copy **B200/GB200/SM100** code paths. NVIDIA’s Blackwell tuning guide distinguishes compute capability **10.0** and **12.0** resource limits: CC 12.0 has **48 resident warps/SM**, **128 KB shared memory/SM**, and **99 KB max shared memory per block**, while CC 10.0 has larger shared-memory limits. ([NVIDIA Docs][6])

## Minimal compile baseline

For a custom CUDA extension/kernel targeting RTX 5060 Ti:

```bash
nvcc -O3 \
  -gencode=arch=compute_120,code=sm_120 \
  -gencode=arch=compute_120,code=compute_120 \
  your_kernel.cu -o your_kernel
```

Including PTX is useful because NVIDIA’s compatibility guide explains that PTX is forward-compatible and can be JIT-compiled when no matching cubin exists, while native cubins avoid runtime JIT cost. ([NVIDIA Docs][8])

## My recommended priority order

Start with **FP4/NVFP4 block-scaled weight GEMM/GEMV**, then fix **memory layout/coalescing**, then tune **shared-memory/register occupancy**, then add **fused epilogues**, then optimize **KV-cache decode**, and only then experiment with **DSM/TMA/cluster-level tricks**. On a 5060 Ti, bandwidth and launch overhead can dominate before the fancy Blackwell features show their benefit.
